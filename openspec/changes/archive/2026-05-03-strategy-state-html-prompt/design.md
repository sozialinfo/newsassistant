## Context

`strategy.strategy` is the central configuration object for the news assistant's AI-driven article labelling. Currently all strategies are treated as perpetually active, and the prompt is stored as plain text generated by the AI (read-only). There is no way to mark a strategy as "in progress" or "retired." The digest and article evaluation pipelines blindly pick up all strategies regardless of readiness.

The codebase already has a pattern for `state` + `statusbar` on `strategy.digest`. The `newsassistant` base module has no shared utilities yet.

## Goals / Non-Goals

**Goals:**
- Introduce draft/active/archived lifecycle on `strategy.strategy`
- Restrict digest and article evaluation to `active` strategies only
- Make the prompt field an HTML field (editable, rich formatting)
- Convert HTML prompt to plain text on the fly before LLM consumption
- Add a shared `html_to_markdown()` utility in the base module
- Consolidate all HTML-stripping currently done ad-hoc with `re.sub`
- Improve the prompt tab UX: rename, move button, add explanation, add overwrite confirmation
- Guard activation behind prompt presence; offer distill wizard when content available

**Non-Goals:**
- Changing the AI distillation logic itself
- Adding audit trail / chatter to `strategy.strategy`
- Supporting non-PDF source documents

## Decisions

### D1: `state` as `Selection` field, not `active`/`archive` mixin

**Choice:** Plain `Selection([draft, active, archived])` field, manually managed.

**Rationale:** Odoo's `mail.thread` + `active` mixin gives automatic archiving but conflates "archived" with "hidden." We need all three states visible in list view and all transitions to be explicit. The `strategy.digest` model uses the same pattern.

**Alternative considered:** `active = fields.Boolean` + separate `state`. Rejected — redundant and confusing.

### D2: All state transitions allowed in both directions

**Choice:** Any state → any state. No one-way gates except the activation guard (prompt required).

**Rationale:** Administrators need flexibility to revert a mistaken activation or reactivate an archived strategy without recreating it.

### D3: Activation guard via wizard, not `@api.constrains`

**Choice:** `action_activate()` method checks for prompt; if missing, opens a wizard (`strategy.activate.confirm`) that offers to distill first or cancel.

**Rationale:** `@api.constrains` raises a `ValidationError` with no recovery path. A wizard can offer an immediate "Distill & Activate" action, which is a much better UX. The guard only applies to `→ active`; other transitions are unconditional.

**Wizard logic:**
- If `prompt` is set → write `state = 'active'` directly, no wizard
- If `prompt` not set AND (description OR document_ids) → open `strategy.activate.confirm` wizard with "Distill & Activate" button
- If `prompt` not set AND no content → raise `UserError` with clear message

### D4: Distill overwrite confirmation via wizard

**Choice:** `action_distill_prompt()` checks if `prompt` is already set; if so, opens `strategy.distill.confirm` wizard. User confirms → distillation runs. Cancel → no-op.

**Rationale:** Prevents accidental overwrite of a carefully edited prompt. The wizard is minimal (no fields, just a confirm button).

### D5: `prompt` changes from `Text(readonly=True)` to `Html()`

**Choice:** `fields.Html()` with no `readonly` — fully editable by users.

**Rationale:** Rich HTML formatting (headings, lists, bold) makes complex prompts more readable and maintainable. The LLM receives plain text anyway (converted on the fly).

**Implication:** `action_distill_prompt()` must write HTML (wrap plain-text AI output in `<p>` tags). Existing plain-text prompts in demo data become valid HTML automatically since Odoo's Html field accepts plain text.

### D6: `html_to_markdown()` utility in `newsassistant/models/utils.py`

**Choice:** Standalone module-level function using `beautifulsoup4` (already installed). Not a mixin method.

**Rationale:** Pure function is easier to test and import. No need for Odoo ORM context. BeautifulSoup4 is already a dependency of the running container.

**Conversion approach:**
- `<p>`, `<br>` → newlines
- `<li>` → `- ` prefix
- `<h1>`–`<h6>` → `#` prefixes
- `<strong>`, `<b>` → `**...**`
- `<em>`, `<i>` → `*...*`
- All other tags → stripped, text preserved
- HTML entities decoded

**Consumers updated:**
1. `strategy_digest.py`: `strategy.prompt` → `html_to_markdown(strategy.prompt)`
2. `news_article.py` (strategy_digest module): `strategy.prompt` in system prompt → `html_to_markdown(strategy.prompt)`; `self.content` strip → `html_to_markdown(self.content)` (replaces `re.sub`)

### D7: Active-only filter in digest and article evaluation

**Choice:** Add `("state", "=", "active")` domain to `strategy.strategy` searches in `strategy_digest._get_active_strategies_for_period()` and `news_article._evaluate_for_strategy()`.

**Rationale:** Draft and archived strategies should not affect production output.

### D8: Statusbar in header + badge in list view

**Choice:** Follow the exact `strategy.digest` pattern:
- Form: `<field name="state" widget="statusbar" statusbar_visible="draft,active,archived"/>`
- List: `<field name="state" widget="badge" decoration-muted="state=='draft'" decoration-success="state=='active'" decoration-danger="state=='archived'"/>`

## Risks / Trade-offs

- **Existing strategies become `draft`** on migration → they will be excluded from digest/evaluation until explicitly activated. Demo data must create strategies in `active` state to avoid breaking smoke tests.
- **HTML prompt storage** means existing plain-text prompts (if any in DB) display correctly in the HTML editor but may lack formatting. Acceptable.
- **BeautifulSoup4 conversion** is a best-effort HTML→text transform. Edge cases (nested lists, tables) will degrade gracefully to plain text — sufficient for LLM consumption.

## Migration Plan

No explicit migration file needed — the `state` field defaults to `draft`. Demo data updated to set `state='active'` on all demo strategies. Fresh instance with demo data validates the full flow.
